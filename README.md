# AI Recruiter v2.0 🧠✨
> **Redrob Intelligent Candidate Discovery Challenge — Hackathon Submission**

AI Recruiter v2.0 is an end-to-end candidate ranking and signal extraction platform. It automates the parsing of unstructured Job Descriptions (JDs) using Google Gemini and evaluates candidates locally using a high-performance semantic scoring engine. By combining deep semantic matching, fuzzy string similarity, and a negative constraint disqualifier system, the platform ranks candidates deterministically in milliseconds—completely eliminating LLM hallucinations and high operational API costs during candidate evaluation.

## 🛠️ Key Engineering Enhancements & Upgrades

During the development cycle, several production-grade improvements were made to address LLM constraints, API rate limits, and scoring accuracy:

### 1. Robust JSON Parsing & Auto-Repair
* **The Problem**: Gemini can hit token limit cutoffs or randomly truncate outputs when asked to produce large JSON blocks, leading to malformed syntax and parsing failures.
* **The Solution**: Implemented a custom `_repair_truncated_json` mechanism in `app/services/extraction_service.py`. It tracks quote states, trims trailing commas, and automatically closes unclosed JSON brackets and braces in real-time, successfully recovering truncated LLM responses.
* **Sanitization Layer**: The `_sanitize_dict` method maps missing/null values into safe defaults (e.g., converting null experience bounds to `0` or null lists to `[]`) to ensure Pydantic schema validation never throws `422 Unprocessable Entity` errors.

### 2. Upstream API Resilience & Fallback Chains
* **Multi-Model Failover**: To mitigate rate limits (HTTP `429 Too Many Requests`) on the Gemini free tier, the system employs an automated fallback list:
  1. `gemini-2.5-flash` (Primary)
  2. `gemini-2.0-flash-lite` (Fallback)
  3. `gemini-2.0-flash` (Fallback)
  4. `gemini-2.5-pro` (Fallback)
* **Backoff Strategy**: Incorporates retry intervals (5s, 10s, 15s) with a 5-second cooldown period between model switches to give the quota window time to clear.

### 3. Three-Layer Semantic & Fuzzy Skill Matching
* **Cosine Similarity**: Uses SentenceTransformers `all-MiniLM-L6-v2` with a similarity threshold of `0.55` to map candidate skills to JD requirements semantically (e.g. aligning "embeddings-based retrieval" with "dense vectors").
* **Fuzzy Match Fallback**: Uses `rapidfuzz` (Token Set Ratio >= 75%) to capture typographical variations, typos, or direct name matches.
* **Summary Analysis**: Performs partial string matching (Partial Ratio >= 80%) on candidate summaries. This combination guarantees candidates are scored fairly and eliminates the "zero-point" core skill issue common in traditional keyword searchers.

### 4. Hard Disqualifiers Safeguard (Negative Constraints)
* **Rule Engine**: Explicitly filters and applies a severe penalty (`-50.0` points) for candidate profiles matching disqualifiers, such as consulting-only backgrounds (e.g. TCS/Infosys/Wipro), pure academic research with no production deployment, and computer vision engineers with no NLP exposure.
* **Fallback Generator**: Implemented `_ensure_disqualifiers` to dynamically scan job descriptions for critical keywords. If the LLM misses them, it automatically inserts standard disqualifiers to safeguard recruitment quality.

### 5. Shortlist Scaling & Cache Management
* **Scale-Up**: The frontend and backend architectures were fully scaled to support scoring up to **10,000 candidates** in a single batch, overriding the previous 50-candidate limit.
* **Thread-Safe Embeddings Cache**: The `CandidateLoaderService` utilizes a thread-safe local cache of candidate skill embeddings. Warm query latency for scoring is **< 10ms** for 10 candidates and **< 100ms** for 50 candidates, since no new external LLM calls are needed during the ranking phase.
* **Cache Management Endpoint**: Added `POST /api/clear-jd-cache` to allow manual cache clearing of extracted JDs directly from the UI or API when JDs are updated.

## ⚖️ Scoring Weights & Heuristics

The system computes scores out of **100 total points**:

| Scoring Signal | Weight | Logic / Heuristic |
|:---|:---:|:---|
| **Must-Have Skills** | `40%` | Extracted must-have skills evaluated via semantic similarity (0.55 threshold) + fuzzy match (75% threshold) fallback. |
| **Experience Fit** | `20%` | Full score if candidate years of experience falls in the range. Penalized proportionally for gaps below the minimum requirement. |
| **Nice-to-Have Skills** | `15%` | Secondary skills matched semantically + fuzzy; acts as an extra tie-breaker score. |
| **Behavioral Signals** | `10%` | Normalized combination of profile completeness, recruiter response rate, interview completion rate, and last active status. |
| **Location Fit** | `10%` | Matches locations; grants full score if remote is OK or if candidate is willing to relocate. |
| **Notice Period** | `5%` | Full score for notice periods ≤ preferred notice days. Penalized if notice is longer. |
| **Disqualifiers** | `-50 / hit` | Deducted for hitting hard disqualifiers (e.g. consulting-only history). |

## 📊 Presentation Reference

These talking points align directly with the official Redrob presentation structure:

### 1. Problem Statement
* Recruitment is slowed down by exact-keyword search engines, leading to high-quality candidates being missed due to semantic mismatches.
* Resume-stuffers exploit simple keyword parsers, while generative LLM ranking is too slow, expensive, and prone to hallucinations for high-scale batch screenings.

### 2. Solution Overview
* **Deterministic Scoring Engine**: Uses SentenceTransformers (`all-MiniLM-L6-v2`) and rapidfuzz locally on CPU, ensuring zero hallucinations and ultra-low latencies (`<100ms` for 50 candidates).
* **Negative Constraint Handling**: Leverages hard disqualifier matches to immediately filter out candidates violating absolute requirements.
* **Premium UX Dashboard**: Features a pastel claymorphic design that shows transparent score breakdowns and automated recruiter notes for quick screening decisions.

### 3. Explainability & Trust
* Deterministic mathematical calculations generate detailed score breakdowns for Must-Have, Nice-to-Have, and Behavioral categories.
* Autogenerated Recruiter Notes explain why a candidate is ranked high or low, giving recruiters actionable signals.
